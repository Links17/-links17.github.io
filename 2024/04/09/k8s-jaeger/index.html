K8s链路追踪预研及部署使用

一、概述

本文是做kubernetes的istio当中的链路追踪的预研，对比其他选型及如何部署使用
根据业务需求对整个链路的检测，本文通过对比常见链路追踪工具skywalking,zipkin,jaeger,选择出jaeger实施链路追踪方案

二、常见链路追踪工具

2.1 Skywalking

Skywalking是Apache开源的分布式链路追踪系统。
Skywalking是APM(应用性能管理)领域的一种技术解决方案，是专门为基于云原生容器的分布式系统，主要用于收集、分析、聚合和可视化来自服务和云原生基础设施的数据。

SkyWalking逻辑上分为四部分: 探针, 平台后端, 存储和⽤户界⾯。 探针：收集从应用采集到的链路数据，并将其转换成Skywalking能够识别的数据格式。
平台后端：支持数据聚合、分析和流处理，包括追踪（Trace）、指标（Metric）和日志（Log）等。
存储：通过开放的、可插拔的接口来存储Skywalking的链路数据。Skywalking目前支持ElasticSearch、H2、MySQL、TiDB、InfluxDB等存储组件。
UI：一个可定制化的基于WEB的界面，允许Skywalking终端用户管理和可视化Skywalking的链路数据。

2.2 zipkin

Zipkin是Twitter开源的分布式链路追踪系统，兼容 OpenTracing API，基于 Google Dapper 的论文设计，能够收集服务间调用的时序数据，提供调用链路追踪的分布式追踪系统。

Zipkin Server包含四个组件： collector、storage、search、web UI
collector： 信息收集器，接受或者收集各个应用传输的数据。 storage：存储组件，zipkin 默认直接将数据存在内存中，还支持Cassandra、ElasticSearch
和 Mysql 等存储组件进行数据的存储。 search：查询组件，提供简单的 JSON API 来供外部调用查询。 web UI: 服务端展示平台，提供简单的
web 界面

2.3 jaeger

Jaeger是Uber开源的分布式链路追踪系统，兼容 Open Tracing API，基于 Google Dapper 的论文设计。

2.3.1 概述

jaeger分为6个模块：Jaeger-client，Agent，Collector，Query，Ingester Jaeger-client：包括含用于分布式跟踪的 OpenTracing API
的特定语言实施。Jaeger客户端代码库，便于不同语言的项目来介入到Jaeger中，当我们的应用程序装载上之后，client会负责收集并发送数据到Agent。
Jaeger-agent：一个网络守护进程，监听通过UDP发送的spans，这些spans被批处理并发送到collector。该代理应与所检测的应用放置在同一主机上。这通常是通过
Kubernetes 等容器环境中的 sidecar 实现的 Jaeger-collector：从agent收集Trace信息，并通过处理管道处理他们，再写入后端存储。
Query：Query查询是一种从存储中检索Trace，并提供UI以显示它们的服务。 Ingester：Jaeger collector的精简版，支持直接将kafka数据写入到数据存储端

三、jaeger部署及使用

3.1 背景

istio的安装包⾥是带有jaeger部署⽅式的，但是他提供的部署⽅式是all-in-one，即数据存储是内存，这对于⽣产环境来说是不合适的，因此，需要根据官⽅的⽣产环境的部署⽅式进⾏部署，这个在istio的官⽅⽂档⾥也有说明

3.2 先决条件

在jaeger1.31版本后jaeger需要使用 webhook 来验证 Jaeger 自定义资源 (CR)，这需要安装cert-managerInstallation - cert-manager
